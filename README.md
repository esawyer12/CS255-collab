# CS255-collab
This repo will serve as my portfolio space for SNHU's course CS255

The DriverPass project involved designing an online learning and scheduling platform for a company called DriverPass, whose goal was to reduce the high failure rate among students taking DMV driving tests. The client wanted a cloud-hosted, web-based system that could serve three distinct user roles such as students, instructors, and administrators. All while offering tiered training packages, automated lesson scheduling, interactive practice exams, and real-time DMV data synchronization. The system also needed to handle student progress tracking, role-based access control, and automated email notifications, all accessible from both desktop and mobile devices.

I believe the strongest part of this work was translating the client's interview transcript into a cohesive, layered design. Rather than treating the requirements as a flat checklist, I organized them into a functional hierarchy. From the use case diagram that mapped every stakeholder interaction, down to the class diagram that gave developers a concrete data model to build from. The sequence and activity diagrams bridged those two levels by showing exactly how data flows and decisions are made at runtime, which makes the design immediately actionable. If I could revise one area, it would be the class diagram's relationship notation. The multiplicity labels and some of the association arrows could be made more precise (for example, explicitly annotating 1-to-many versus many-to-many cardinalities between Student, Appointment, and Vehicle) to leave less ambiguity for developers.

Interpreting user needs meant staying grounded in what DriverPass's actual problem was: students were failing tests because they lacked structured preparation and easy access to lessons. Every design decision was filtered through that lens. For example, the conflict-checking logic in the scheduling activity diagram directly addresses the client's need to prevent double-bookings and is not just as a technical feature, but as something that would erode student trust if it failed. Considering user needs during design is critical because a technically correct system that frustrates or confuses its users will not be used, which means the problem the system was built to solve remains unsolved. Going forward I would more deliberately incorporate iterative reviews, presenting early drafts of diagrams for feedback before building out full documents. Since catching a misunderstood requirement early on is far less costly than discovering it in the class or sequence diagram creation.
